## Apps Script Architecture

The dashboard is too large (~970KB) for Apps Script's HtmlService, so we use an **async chunked loading** pattern:

1. `doGet()` serves `Dashboard.html` (~263KB, HTML/CSS shell only) via `createHtmlOutputFromFile`
2. Client-side loader calls `google.script.run.getDashboardJSChunk(0..3)` in parallel
3. JS chunks are stored as **string constants in `.gs` files** (`_JS_CHUNK_0`, `_JS_CHUNK_1`, etc.) — this bypasses HtmlService entirely, which corrupts JS containing HTML-like patterns (innerHTML, template literals with HTML tags, etc.)
4. Each chunk is <200KB (under google.script.run's ~256KB return limit)
5. Once all chunks arrive, they're concatenated and injected as a `<script>` element
6. The init function checks `document.readyState` to handle the fact that `DOMContentLoaded` has already fired by the time injected JS executes
7. Data fetching routes through `google.script.run.proxyFetch(url)` → server-side `UrlFetchApp` to bypass iframe CSP

### Why .gs files instead of .html for chunks

Previous attempts stored JS chunks as `.html` files and read them via `HtmlService.createHtmlOutputFromFile().getContent()`. This failed because HtmlService processes the content as HTML, corrupting JS that contains HTML-like patterns (which this dashboard has extensively — innerHTML assignments, template literals with `<div>`, `<table>`, etc.). The `.gs` approach stores the JS as a JSON-escaped string variable, completely bypassing HTML processing.

## Deploy Pipeline

**GitHub Pages:** Automatic via `.github/workflows/deploy.yml` on push to main.
- Scoped to only publish `index.html`, `dashboard.js`, and `coord-history.json` (not the entire repo root)
- This prevents exposing `.clasp.json`, `.gs` files, `push.command`, and other internal files on the public Pages site

**Apps Script:** Automatic via `.github/workflows/deploy-appscript.yml`:
1. Triggered on push to main (paths: index.html, dashboard.js, appscript/**, scripts/**, the workflow itself)
2. Runs `python3 scripts/build-appscript.py` to generate Dashboard.html + `.gs` JS chunks
3. Validates clasp credentials (JSON parse check before push)
4. `clasp push --force` uploads to Apps Script project
5. `clasp deploy -i <deployment_id>` updates the existing web app deployment (not creating new ones)

**Secret required:** `CLASPRC_JSON` in GitHub repo settings (OAuth credentials from `~/.clasprc.json`). Must be set via `gh secret set CLASPRC_JSON < ~/.clasprc.json` — the JSON content is written using `printenv` to avoid shell escaping issues.

## Data Sources

All data is fetched from Google Sheets published as CSV. On GitHub Pages, standard `fetch()` is used. On Apps Script, fetches route through `google.script.run.proxyFetch()` using the `IS_APPS_SCRIPT` detection flag (line ~10048 in dashboard.js).

## localStorage Keys

- `crp_coord_snapshot_v1` — Coordinator visit data snapshot
- `crp_dismissed_actions_v1` — Dismissed action items (60-day auto-prune)
- `crp_coll_[invoice]` — Invoice collection status/notes
- `crp_clickup_token` — ClickUp API token (user-entered)
- `crp_fin_auth` (sessionStorage) — Finance PIN auth flag

## Coordinator Visit History

Three-layer merge system (highest count wins):
1. **coord-history.json** (permanent baseline, committed to repo)
2. **localStorage snapshot** (recent browser-cached data)
3. **Live CRIO data** (fetched at runtime)

Export button (⬇) in Coordinator Visit Snapshot header downloads merged history as JSON.

## New Machine Setup

1. Clone repo
2. `npm install -g @google/clasp`
3. `clasp login` (opens browser for Google OAuth — use `@phillyresearch.com` account)
4. Copy `.clasp.json` to `appscript/` with the Script ID above
5. Update `CLASPRC_JSON` secret: `gh secret set CLASPRC_JSON --repo aneesh-CRP/gh-repo-create-crp-unified-dashboard---public < ~/.clasprc.json`
6. Everything else auto-deploys via GitHub Actions

## Code Review Findings (March 12, 2026)

Full code review was performed. Key findings by severity:

### Fixed in this session
- **Hardcoded dates**: All `new Date('2026-03-07')` replaced with `new Date()` — schedule filters, action items, and sort comparisons now use live dates
- **Invalid CSS**: `flex-wrap: gap` → `flex-wrap: wrap` in setup footer
- **GitHub Pages leaking files**: Deploy scoped to dashboard files only
- **Apps Script deploy creating orphan deployments**: `clasp deploy` now updates existing web app deployment ID
- **CLASPRC_JSON secret malformed**: Re-set from local credentials using `gh secret set`
- **Apps Script credential writing**: Changed from `echo` to `printenv` to avoid shell escaping issues with JSON
- **DOMContentLoaded timing**: Init function now checks `document.readyState` for async chunk injection

### Outstanding (not yet fixed)
- **PHI in source code**: Patient names, visit dates, cancellation reasons hardcoded in inline JSON (~6K lines). Should be fetched at runtime behind authentication.
- **SSRF in proxyFetch**: `appscript/WebApp.gs` accepts any URL with zero validation. Should whitelist allowed domains.
- **98 innerHTML assignments**: Data from Google Sheets/ClickUp interpolated without sanitization. XSS risk if data sources compromised.
- **ClickUp token in localStorage**: Accessible to any JS on the same origin.
- **Client-side PIN auth**: Bypassable via `sessionStorage.setItem('crp_fin_auth','true')`. All finance data is in page source regardless.
- **stageColors defined 7 times**: Inconsistent values across functions.
- **Duplicate functions**: `sortTable`/`sortDetailTable`, `drawRevChart`/`drawRevChartOn`, `drawPayChart`/`drawPayChartOverview`.
- **O(n*m) patient lookups**: `.find()` in loops — should use Map.
- **Auto-refresh on tab switch**: No cooldown despite comment claiming 5-min check.
- **CRP_CONFIG.REFRESH_INTERVAL ignored**: 15-min interval hardcoded.
- **Stale files**: Root-level `WebApp.gs` and `finance_consolidator.gs` (template) are outdated duplicates.
- **push.command**: Enables one-click push to main bypassing review.
